# WitDB: требования и план реализации чисто .NET файловой СУБД

## Предпосылки и цели проекта

**WitDB** – проект встраиваемой СУБД, нацеленной на то, чтобы стать **аналогом SQLite в экосистеме .NET**. В отличие от SQLite, написанной на С и используемой через нативную библиотеку, WitDB планируется реализовать целиком на управляемом коде .NET (C#). Это устранит сложности с интеграцией нативных DLL (особенно на платформах вроде iOS, WASM и т.д.) и обеспечит бесшовную работу во всех средах .NET[tomaskohl.com](https://tomaskohl.com/code/2020-04-07/trying-out-litedb/#:~:text=Because%20SQLite%20is%20written%20in,dotnet%20code)[tomaskohl.com](https://tomaskohl.com/code/2020-04-07/trying-out-litedb/#:~:text=LiteDB%20is%20100,this%20is%20hard%20to%20overstate). В настоящее время разработчикам .NET не хватает подобного решения: единственная полноценная файло-ориентированная СУБД для .NET – SQLite – **не является “pure .NET”** и требует внешней библиотеки. Существующие чисто .NET решения, такие как LiteDB, тоже не полностью закрывают потребности: LiteDB – документно-ориентированная NoSQL-база, не поддерживающая всех возможностей SQL (например, **отсутствуют JOIN и строгая схема**)[tomaskohl.com](https://tomaskohl.com/code/2020-04-07/trying-out-litedb/#:~:text=Unlike%20SQLite%2C%20LiteDB%20is%20a,document%20database). Кроме того, LiteDB не имеет готового провайдера для Entity Framework Core, а другие варианты (например, устаревший SQL Server Compact или коммерческий VistaDB) либо ограничены платформенно, либо не поддерживают EF Core[doc.vistadb.com](https://doc.vistadb.com/EntityFramework_Introduction.html#:~:text=What%20About%20Entity%20Framework%20Core%3F)[reddit.com](https://www.reddit.com/r/dotnet/comments/5ututm/looking_for_an_embedded_db_that_isnt_sqlite_for/#:~:text=match%20at%20L221%20Was%20gonna,the%20bullet%20on%20managing%20foreign). **Цель WitDB** – заполнить эту нишу, предоставив **кроссплатформенную, легковесную и полностью управляемую СУБД** с поддержкой **Entity Framework Core**. Ниже сформулированы требования к WitDB и приведён план её технической реализации.

## Требования к WitDB

### Функциональные возможности

- **Полноценная реляционная СУБД с SQL-интерфейсом.** WitDB должна поддерживать стандартный SQL для операций **DDL** (создание/изменение схемы) и **DML** (запросы, вставка, обновление, удаление). Как минимальное требование – реализовать основные конструкции SQL-92/99, включая `SELECT` с `JOIN`, вложенные запросы, агрегатные функции, **транзакции** и т.д. Если SQLite что-то поддерживает из SQL, то и WitDB должна стремиться это поддерживать[geeksforgeeks.org](https://www.geeksforgeeks.org/sql/introduction-to-sqlite/#:~:text=2,the%20direct%20file%20system%20I%2FO). В перспективе планируется реализация современных возможностей SQL, известных по SQLite: например, **common table expressions (CTE)**, оконные функции, частичные и экспрессионные индексы, встроенные функции работы с JSON и т.д.[geeksforgeeks.org](https://www.geeksforgeeks.org/sql/introduction-to-sqlite/#:~:text=2,easy%20to%20port%20to%20other).
    
- **Совместимость с Entity Framework Core.** Необходимо разработать ADO.NET-провайдер и провайдер для EF Core, позволяющие использовать WitDB через `DbContext` EF Core аналогично тому, как используется SQLite. Это означает поддержку **LINQ-запросов** (т.е. перевод их в SQL, понятный движку WitDB) и основных функций EF Core – **миграции схемы, привязка навигационных свойств, отслеживание изменений** и пр. Провайдер должен обеспечить выполнение команд создания/изменения таблиц, необходимых для `DbContext.Database.Migrate()` без ошибок (например, поддержка операций изменения столбцов или удаления ограничений, с которыми у SQLite возникают сложности). Таким образом, разработчики смогут легко переключаться между SQLite и WitDB или использовать WitDB прозрачно для приложений, написанных под EF Core.
    
- **Serverless-архитектура, встраивание в приложение.** По аналогии с SQLite, база данных WitDB хранится в **одном файле** на диске (плюс возможно вспомогательный файл журнала транзакций). Не требуется запуск отдельного сервера – библиотека работает в том же процессе, что и приложение (**embedded DB**). Достаточно подключить сборку (NuGet-пакет) – никаких внешних зависимостей или установки СУБД не нужно[tomaskohl.com](https://tomaskohl.com/code/2020-04-07/trying-out-litedb/#:~:text=Just%20like%20SQLite%2C%20LiteDB%3A).
    
- **ACID-транзакции и целостность данных.** WitDB обязана обеспечивать _атомарность, консистентность, изолированность и долговременность_ транзакций на уровне не ниже SQLite. Все операции изменений должны быть транзакционными с возможностью отката, а база не должна повреждаться при сбоях (сохранность данных даже в случае сбоя питания или краха приложения)[geeksforgeeks.org](https://www.geeksforgeeks.org/sql/introduction-to-sqlite/#:~:text=1,the%20direct%20file%20system%20I%2FO). Будет реализована журнальная система (см. ниже) для гарантии долговременности. Также планируется поддержка **ограничений целостности**: первичные и внешние ключи (аналогично SQLite, где FOREIGN KEY включаются pragma-параметром), уникальные ограничения, проверки (`CHECK`). WitDB должна корректно применять и соблюдать ограничения (то есть, например, отклонять вставку, нарушающую FOREIGN KEY). Если SQLite умеет поддерживать целостность через ограничения или триггеры, WitDB должна предоставлять аналогичный функционал.
    
- **Отсутствие искусственных ограничений и масштабируемость.** По требованию заказчика, "особых ограничений не было": это означает, что WitDB не должна вводить жестких лимитов на размеры и состав БД сверх разумных пределов и возможностей файловой системы. Планируется поддержка файлов БД **большого размера** – вплоть до гигабайт и терабайт, как это делает SQLite[geeksforgeeks.org](https://www.geeksforgeeks.org/sql/introduction-to-sqlite/#:~:text=expressions,platform%20disk%20file). Также не ограничивается число таблиц, индексов, соединений – они могут определяться лишь ресурсами окружения (памятью, диском). Количество столбцов в таблице, длина строк, размер BLOB/текстовых полей – все аналогично SQLite (например, SQLite допускает до 32 000 столбцов в таблице и многогигабайтные BLOB)[geeksforgeeks.org](https://www.geeksforgeeks.org/sql/introduction-to-sqlite/#:~:text=expressions,platform%20disk%20file). WitDB должна работать производительно и корректно как с маленькими, так и с крупными базами.
    
- **Кроссплатформенность и универсальность для .NET.** WitDB нацелена на работу **во всех сценариях .NET**: классические десктоп-приложения (Windows, Linux, macOS), мобильные приложения (Xamarin/MAUI для iOS, Android), приложения на **Blazor WebAssembly** (WASM), Unity и проч. Для этого ядро базы будет написано на .NET (без P/Invoke), что позволит запускать ее даже там, где использование нативных библиотек затруднено. Например, в Blazor WASM нельзя свободно загрузить нативную .dll, поэтому наличие чисто .NET реализации дает огромное преимущество. WitDB должна быть совместима с .NET 8/9 и выше, ориентируясь на .NET Standard 2.1+ для максимального охвата. Разумеется, вся функциональность (доступ к файлам, многопоточка) будет аккуратно использоваться с учетом разных платформ (напр., в WASM нет привычной файловой системы, но можно хранить данные в IndexedDB – см. ниже об абстракции хранения).
    
- **Поддержка обоих режимов хранения: файлового и полностью в памяти.** Как и SQLite, WitDB должна уметь работать **в памяти** без сохранения на диск, если это нужно (например, для быстрых вычислений, временных данных или в среде без дисковой подсистемы). Специальный режим in-memory активируется через строку подключения (по аналогии с `Data Source=:memory:` у SQLite[sqlite.org](https://sqlite.org/inmemorydb.html#:~:text=The%20most%20common%20way%20to,For%20example)) или программно. В этом режиме все данные хранятся в оперативной памяти и исчезают при закрытии соединения. Также необходим вариант **персидентного хранения** в файле на диске. Таким образом, разработчик выбирает нужный вариант (в памяти для тестов или временных расчетов, либо файл для долговременного хранения) – **обе опции должны поддерживаться** (требование заказчика) из коробки.
    
- **Многопользовательский доступ и параллелизм.** WitDB будет допускать одновременную работу нескольких потоков (или даже процессов) с одной базой, аналогично модели SQLite. Требование “если SQLite поддерживает, то и она должна” здесь означает: **несколько одновременных читателей** и как минимум **один писатель за раз** без нарушений целостности данных[blog.skypilot.co](https://blog.skypilot.co/abusing-sqlite-to-handle-concurrency/#:~:text=,to%20seek%20a%20different%20solution). В SQLite при использовании режима журнала WAL множество потоков могут читать базу параллельно во время записи, но запись все равно выполняется по одной транзакции за раз (параллельные записи в одну базу не выполняются одновременно)[blog.skypilot.co](https://blog.skypilot.co/abusing-sqlite-to-handle-concurrency/#:~:text=SQLite%20can%20also%20run%20in,writers%20until%20it%20is%20finished). WitDB реализует такую же модель: транзакции записи будут блокировать запись для других до завершения, однако чтение не будет блокироваться (при использовании механизма WAL). Это достаточно для большинства сценариев; приложение же со своей стороны должно позаботиться о корректной синхронизации обращений или использовать очереди запросов, если пишущих потоков много. В дальнейших планах можно рассмотреть расширения для улучшения конкурентности (например, оптимистичные транзакции ala `BEGIN CONCURRENT` SQLite[stackoverflow.com](https://stackoverflow.com/questions/4060772/sqlite-concurrent-access#:~:text=BEGIN%20CONCURRENT%20,detailed%20doc)), но базовое требование – не хуже SQLite по многопоточности. Также WitDB должна работать корректно при открытии одной базы в нескольких процессах (через механизм файловых блокировок), хотя основной сценарий – множественные подключения внутри одного приложения.
    
- **Высокая надёжность и тестируемость.** Будучи встроенной СУБД, WitDB должна быть **очень надёжной**, так как любая ошибка движка может привести к потере данных пользователя. Требуется обеспечить детальное тестирование: функциональные тесты SQL-операций, интеграция тестов на совмещение с EF Core, проверки на потокобезопасность (имитация конкурентных транзакций), тестирование отказоустойчивости (эмулировать сбои питания и проверять целостность файла). Репутация SQLite во многом строится на многолетнем тщательном тестировании и проверках[tomaskohl.com](https://tomaskohl.com/code/2020-04-07/trying-out-litedb/#:~:text=SQLite%20has%20long%20been%20the,well%20when%20used%20as%20intended) – WitDB должна стремиться к подобному уровню уверенности в корректности. Это подразумевает также четкую спецификацию формата файла и поведения, чтобы внешние разработчики могли понять гарантии системы.
    
- **Производительность и оптимизация под .NET.** Несмотря на управляемую природу, производительность WitDB должна быть как минимум приемлемой в сравнении с SQLite. Будут использоваться возможности .NET для ускорения работы (см. раздел архитектуры и оптимизаций ниже). Цель – обеспечить быстрые операции для типичных рабочих нагрузок: точечные выборки по ключу, сканирование небольших таблиц, вставка/обновление пачек записей – все это должно выполняться за миллисекунды. Конечно, SQLite, написанный на C, может быть быстрее на некоторых операциях, но с учётом постоянного улучшения JIT/AOT в .NET разница не должна быть критичной. Важно также обеспечить **низкую нагрузку на память** (работать на устройствах с ограниченной RAM) и **малый размер библиотеки**. Планируется держать размер сборки WitDB небольшим (порядка нескольких сотен KB или ниже пары MB), чтобы её можно было без раздумий включать даже в мобильное приложение (для сравнения, одна .dll LiteDB ~450KB[litedb.org](https://www.litedb.org/#:~:text=LiteDB%20%3A%3A%20A%20,0)). Кроме того, требуется избегать операций, вызывающих частую сборку мусора или паузы, чтобы не деградировала отзывчивость UI в UI-приложениях. Для этого будут использованы такие подходы, как объектные пулы, структуры вместо классов для часто создаваемых сущностей и т.д.
    
- **Безопасность и расширяемость.** WitDB должна поддерживать шифрование базы данных (опционально) – например, возможность задать пароль для файла, чтобы данные в файле хранились в зашифрованном виде. SQLite не включает шифрование по умолчанию (требуются внешние решения), но в экосистеме .NET это востребовано, особенно для мобильных приложений. Поэтому в требования можно добавить **опцию шифрования** (возможно, используя алгоритм AES с ключом из пароля). Также неплохо предусмотреть механизм **расширяемости**: например, регистрация пользовательских функций (аналогично `sqlite3_create_function` в SQLite) для использования в SQL-запросах. В .NET это можно сделать, позволив пользователю предоставить делегат C# в качестве реализации новой SQL-функции, которую движок сможет вызывать во время выполнения запроса. Это не первоочередная функция, но желательная для паритета с SQLite.
    

### Нефункциональные требования

- **Совместимость с AOT и ограниченными средами.** Код WitDB должен работать в условиях трейминга и AOT-компиляции (например, на iOS, WASM), поэтому следует избегать динамического генерирования кода во время выполнения (Reflection.Emit, генерация IL) и избыточного использования рефлексии. Все функции должны корректно переживать Linking (удаление неиспользуемого кода) – для этого публичные API должны быть явно помечены как используемые, либо следует предоставлять ссылочную сборку для RyuJIT/AOT. Также необходимо учитывать ограничения полноты .NET API на разных платформах – например, в Unity может быть недоступна `System.Data.Common` (EF Core там не работал до недавнего времени), на WASM нет потока и прямого доступа к диску (будет использоваться эмуляция хранилища – IndexedDB). Эти моменты должны учитываться при проектировании.
    
- **Простота использования.** Интерфейс подключения к базе – знакомый разработчикам .NET. Предполагается, что работа с WitDB будет выглядеть аналогично работе с другими ADO.NET-провайдерами. Например: `using var conn = new WitDbConnection("Data Source=database.witdb"); conn.Open(); ...`. Также планируется поддержка стандартных механизмов конфигурации, таких как **строки подключения** (с параметрами вроде `Cache=Shared` для in-memory, `Mode=ReadWriteCreate` и т.п., по аналогии с Microsoft.Data.Sqlite). Документация должна включать понятные примеры, чтобы порог входа был минимальным.
    
- **Open-Source и лицензирование.** (Если этот аспект важен заказчику.) SQLite – общественное достояние, LiteDB распространяется под LGPL (насколько известно). WitDB желательно сделать открытым проектом с либеральной лицензией (MIT или Apache) для максимального распространения. Это не техническое требование, но организационное, которое может повлиять на принятие решения о внедрении WitDB в реальных проектах.
    

## Архитектура и план технической реализации

Проектирование WitDB разделим на несколько уровней: **движок хранения (storage engine)**, **исполнитель запросов (query engine)** и **слой интеграции с .NET/EF**. Также рассмотрим использование современных оптимизаций .NET при реализации. Ниже описаны основные компоненты и этапы разработки.

### 1. Формат файла и движок хранения данных

**Структура файла.** Подобно SQLite, база данных WitDB будет представлять собой **страничное хранилище** – файл, разбитый на страницы фиксированного размера (например, 4096 байт по умолчанию)[fly.io](https://fly.io/blog/sqlite-internals-btree/#:~:text=SQLite%20Internals%3A%20Pages%20%26%20B)[fly.io](https://fly.io/blog/sqlite-internals-btree/#:~:text=This%20constrained%20size%20means%20that,easy%20for%20anyone%20to%20understand). В начале файла размещается заголовок с идентификатором формата и служебной информацией (версия, размер страницы, кодировка и т.п.). Данные таблиц и индексов хранятся в страницах, организованных в **B-деревья (B+tree)** – это проверенная структура данных, обеспечивающая эффективный поиск по ключу и последовательное сканирование[fly.io](https://fly.io/blog/sqlite-internals-btree/#:~:text=SQLite%20Internals%3A%20Pages%20%26%20B,pages%20for%20efficient%20data%20access)[fly.io](https://fly.io/blog/sqlite-internals-btree/#:~:text=This%20constrained%20size%20means%20that,easy%20for%20anyone%20to%20understand). В WitDB каждая таблица будет храниться в своём B-дереве (ключи которого – идентификаторы строк или значения Primary Key, а значения – сами записи), а каждый индекс – в отдельном B-дереве (где ключи – значение индексируемого поля(полей), а значение – ссылка на соответствующую строку таблицы). Такое разделение аналогично механизму SQLite[jvns.ca](https://jvns.ca/blog/2014/10/02/how-does-sqlite-work-part-2-btrees/#:~:text=How%20does%20SQLite%20work%3F%20Part,data%20is%20stored%20in).

 

**Формат записей.** Каждая строка таблицы кодируется последовательностью байт по определенному формату. Можно взять за основу формат **SQLite record** – он очень компактный. В начале записи идёт заголовок с длинами/типами полей, затем подряд поля. SQLite использует **вариантное целочисленное кодирование (varint)** для экономии места: целые числа хранятся не фиксированными 8 байт, а переменным числом байт, достаточным для их величины[fly.io](https://fly.io/blog/sqlite-internals-btree/#:~:text=These%20first%20two%20fields%20use,gets%20its%20own%20container%20size). WitDB будет применять аналогичный подход: типы данных (INT, TEXT, BLOB, REAL, BOOL, NULL и др.) могут иметь префиксный код, а сами значения хранятся компактно (например, 0 или 1 байт для NULL/Bool, varint для чисел, сжатое представление длины перед строками/блобами). Это позволит экономить место и ускорять обработку (меньше объем IO)[fly.io](https://fly.io/blog/sqlite-internals-btree/#:~:text=These%20first%20two%20fields%20use,gets%20its%20own%20container%20size)[fly.io](https://fly.io/blog/sqlite-internals-btree/#:~:text=Varints%20use%20a%20simple%20trick,split%20into%207%20bit%20chunks).

 

**Управление страницами и памятью.** Движок хранения включает **менеджер страниц**, ответственный за выделение новых страниц, переиспользование освобождённых и кэширование часто используемых страниц в памяти. Планируется реализовать **буферный кэш** с LRU или аналогичным алгоритмом: часто читаемые страницы держать в памяти, чтобы снизить обращения к диску. Размер кэша может быть конфигурируемым (аналогично `PRAGMA cache_size` в SQLite). Для доступа к содержимому страниц активно используем типы `Span<byte>`/`Memory<byte>` из .NET – это позволит работать с байтовыми буферами без лишних аллокаций в куче и копирований. Например, можно читать страницу с диска сразу в арендованный буфер из пула (ArrayPool), и затем передавать Span<byte> в методы парсинга записей. Такой подход минимизирует нагрузку на GC и ускоряет операции ввода-вывода.

 

**Алгоритмы B-дерева.** Реализация B+tree включает операции поиска по ключу, последовательного обхода (итератора), вставки и удаления. Нужно продумать алгоритмы балансировки дерева при вставке/удалении (сплит страниц, слияние страниц). Это достаточно сложная часть, но можно опереться на открытые описания (например, **алгоритм B-деревьев SQLite** подробно описан в их документации[jvns.ca](https://jvns.ca/blog/2014/10/02/how-does-sqlite-work-part-2-btrees/#:~:text=How%20does%20SQLite%20work%3F%20Part,data%20is%20stored%20in)). Особое внимание – на **разделение больших записей**: если запись не помещается в одну страницу, придётся реализовать механизм **overflow pages** (как в SQLite), где длинные BLOB или TEXT распределяются по цепочке страниц. WitDB должен уметь хранить большие объёмы данных в одном поле (вплоть до гигабайтов, хотя в практике это редкость).

 

**Журналирование и транзакции.** Для обеспечения атомарности и долговременности потребуется механизм журнала. Рассматриваются два варианта: **rollback journal** (журнал отката, как в традиционном режиме SQLite) и **WAL (Write-Ahead Logging)**. Второй вариант предпочтителен для современных приложений, так как **WAL позволяет параллельно читать во время записи** и обычно быстрее на смешанных нагрузках[blog.skypilot.co](https://blog.skypilot.co/abusing-sqlite-to-handle-concurrency/#:~:text=SQLite%20can%20also%20run%20in,writers%20until%20it%20is%20finished). Поэтому план: реализовать режим WAL по умолчанию. Это означает, что изменения транзакции сначала записываются в отдельный журнал-файл (WAL), а основная база не блокируется для чтения. При коммите страницы из WAL переносятся в основной файл (или сливаются при чекпоинте). Нужно реализовать формат WAL (можно упростить относительно SQLite, но суть та же) и механизм **checkpoint** – сброса накопившихся изменений в основной файл. При открытии базы, если обнаружен незакрытый журнал, следует откатить или дохватить изменения (recovery). WAL-механизм обеспечит, что база **не испортится** при крахе посреди транзакции: либо изменения полностью применены, либо полностью отвергнуты[geeksforgeeks.org](https://www.geeksforgeeks.org/sql/introduction-to-sqlite/#:~:text=1,the%20direct%20file%20system%20I%2FO). Для простоты первый вариант реализации может быть классический журнал (писать копии изменяемых страниц во временный файл, а потом обратно) – он проще, но тогда чтения блокируются на время транзакции. Однако с учётом требования конкуррентности, лучше сразу спроектировать WAL. Возможно, имеет смысл оставить возможность переключения режима (как `PRAGMA journal_mode` в SQLite, например `WAL` или `ROLLBACK`) для совместимости со сценариями (например, WAL не работает на сетевых шарах, там можно откатиться на rollback-журнал)[stackoverflow.com](https://stackoverflow.com/questions/4060772/sqlite-concurrent-access#:~:text=available%20in%20separate%20branches)[stackoverflow.com](https://stackoverflow.com/questions/4060772/sqlite-concurrent-access#:~:text=,while%20the%20other%20is%20checkpointed).

 

**Блокировки и concurency control.** WitDB должен предотвращать конфликтующий доступ. Понадобится механизм **блокировок на уровне файла и на уровне страницы**. Например, для реализации транзакции чтения/записи можно использовать **файловую блокировку** (FileStream.Lock или аналог) для синхронизации процессов, а внутри процесса – объекты синхронизации (например, `SemaphoreSlim` или `Monitor`) для очереди доступа. В SQLite действует схема уровней блокировки (Shared, Reserved, Exclusive locks)[stackoverflow.com](https://stackoverflow.com/questions/4060772/sqlite-concurrent-access#:~:text=Yes%2C%20SQLite%20handles%20concurrency%20well%2C,html); мы можем реализовать упрощённо: транзакция чтения берёт shared-блокировку файла, транзакция записи – exclusive-блокировку. В WAL-режиме немного иначе (там writer блокирует запись, но не мешает читателям). Это нужно детально спроектировать. Желательно также учесть таймаут ожидания блокировки (например, SQLite по умолчанию ждёт 5 сек перед ошибкой “database is locked”[blog.skypilot.co](https://blog.skypilot.co/abusing-sqlite-to-handle-concurrency/#:~:text=If%20you%E2%80%99re%20getting%20the%20,these%20potential%20causes%20before%20proceeding)). Мы сделаем параметр timeout для попытки начать транзакцию, чтобы при конкуренции писателей один ждал другого некоторое время.

 

**Абстракция хранилища.** Для обеспечения работы в разнообразных средах, слой хранения файлов будет спроектирован абстрактно. Будет некий интерфейс, например `IStorage`, с операциями: открыть файл, прочитать страницу, записать страницу, синхронизировать на диск, установить размер. Обычная реализация будет обёрткой над файловой системой (FileStream с direct access). Но для WebAssembly, где прямого доступа к диску нет, можно предоставить альтернативную реализацию, использующую **IndexedDB** или **MemoryStream** под капотом. Таким образом, сам движок B-деревьев и SQL не зависит от конкретного способа хранения – он работает через интерфейс. Это позволит, например, хранить данные в памяти в сценарии Blazor (с возможностью сохранения снимка базы через Serializing IndexedDB, возможно). Аналогичный подход и у SQLite: у него есть абстракция VFS (virtual file system), мы реализуем что-то подобное, но на уровне .NET.

### 2. Механизм выполнения запросов (SQL-движок)

**SQL-парсер.** Необходим компонент, преобразующий текст SQL-запроса в внутреннее представление (Abstract Syntax Tree, AST или подобие). Мы можем использовать готовые решения, например, **ANTLR** с грамматикой SQL, либо написать собственный парсер (сложнее, но дает полный контроль). Грамматика SQLite SQL достаточно объемна (поддерживает много синтаксиса), но на первом этапе можно ограничиться основным подмножеством: `SELECT ... FROM ... WHERE ...`, `INSERT`, `UPDATE`, `DELETE`, `CREATE TABLE`, `CREATE INDEX`, `DROP`, `ALTER` и пр. Парсер разберет запрос и передаст его планировщику. В дальнейших планах – поддержка `VIEW`, `TRIGGER` (триггеры можно реализовать, храня их как скрытые функции, вызываемые при событиях).

 

**Оптимизатор и планировщик запросов.** На основе AST запросов SELECT/UPDATE и пр., система должна составить план выполнения. На начальном этапе можно реализовать **простой планировщик** с фиксированными стратегиями (например, использование индекса если есть условие по индексу, иначе полное сканирование). SQLite известен продвинутым оптимизатором, учитывающим статистику, перестановки JOIN-ов, индексный охват и т.д. Мы можем упростить: например, реализовать nested-loop join, выполняющийся в порядке написания таблиц в FROM (или с учетом простейшей эвристики – меньшая таблица первой). Индексы будут использоваться для условий равенства (`WHERE Table.Column = ?`) – поиск по B-дереву, для диапазонных условий (`BETWEEN`, `>=`) – также B-дерево позволяет. При отсутствии подходящего индекса – выполнять полное сканирование таблицы. На первом этапе это допустимо. В дальнейшем можно добавить сбор статистики ANALYZE и более умный выбор плана.

 

**Виртуальная машина или интерпретатор.** SQLite реализует исполнитель запросов как стековую виртуальную машину, исполняющую серию инструкций байткода. Мы можем пойти другим путём, ближе к .NET: преобразовать запрос в набор итераторов (итератор таблицы, фильтрация, агрегирование). Например, `SELECT ... FROM A JOIN B WHERE condition` можно реализовать как: для каждой строки из A (итератор по A) вложенно пройти по подходящим строкам из B (итератор по B с условием, возможно индекс). Это паттерн **iterator (cursor)**, часто используемый в движках БД. Мы напишем набор классов-итераторов: `TableScanIterator`, `IndexSeekIterator`, `FilterIterator`, `JoinIterator`, `AggregationIterator` и т.д. Планировщик связывает их в дерево. Выполнение запроса будет происходить путем вызова `MoveNext()` у корневого итератора, который будет запрашивать данные у дочерних и так далее. Такой подход хорошо ложится на C# (можно использовать `IEnumerable` или просто свои методы Next). Альтернативно, можно генерировать LINQ-выражение и компилировать его в делегат – но это сложнее отладочно и может быть тяжело для AOT. Лучше начать с интерпретируемого подхода.

 

**Работа с типами и значения.** Поскольку .NET – типизированная платформа, надо решить, как представлять SQL-значения во время исполнения. SQLite применяет динамическую типизацию "type affinity" – т.е. хранит значения вариантного типа (integer, real, text etc.) и может в любой момент конвертировать. Мы можем сделать подобное: внутри engine иметь тип наподобие `SqlValue` (структура, хранящая тип и значение как union: long, double, string, byte[] и т.д.). Это упростит обработку NULL (отдельный тип) и взаимодействие с .NET (например, `SqlValue` -> `DbDataReader.GetValue()` вернёт типичный .NET-тип). Конверсию типов (например, сравнение строки с числом) тоже реализуем, опираясь на правила SQLite (строка сравнивается как число, если можно). Поддерживаем базовые типы: NULL, INTEGER (64-bit), REAL (double), TEXT (string UTF-16 или UTF-8 внутренняя, но при выдаче как string), BLOB (byte[]), NUMERIC (Decimial) – последний опционально, SQLite не имеет Decimal, но EF Core может ожидать decimal для точных чисел, так что, возможно, реализуем decimal храня его как текст или 128-bit integer.

 

**Схема и метаданные.** Движок должен хранить метаданные о таблицах (список таблиц, колонок, типы, ограничения). В SQLite для этого существует скрытая таблица `sqlite_master`. Мы сделаем аналог – например, служебную таблицу _WitDb_Catalog_ или просто Master, которая хранит определения всех пользовательских объектов (таблиц, индексов, представлений, триггеров). `CREATE TABLE` будет вносить запись в эту таблицу и создавать соответствующие структуры (B-дерево для данных и для каждого индекса). `DROP TABLE` – удалять все связанные страницы обратно в пул свободных и запись из каталога. **ALTER TABLE** – более сложная операция: SQLite ограниченно поддерживает `ALTER TABLE ... RENAME` и `ADD COLUMN`. Мы постараемся поддержать и **удаление столбца, изменение типа столбца** – даже если придется внутренне реализовать как создание новой таблицы и перенос данных (это то, что делает Entity Framework для SQLite). Таким образом, "без ограничений" в требованиях подразумевает, что разработчик через EF Core сможет выполнить любую миграцию, и провайдер WitDB сумеет её применить.

 

**Ограничения и триггеры.** Для FOREIGN KEY планируем хранить их в метаданных и при изменении данных проверять (например, при удалении родительской записи – проверять наличие дочерних, либо реализовать каскадное удаление, если задано). Можно либо сгенерировать скрытые триггеры для enforcement, либо в самом движке проверять при `DELETE/UPDATE`. SQLite включает механизм FOREIGN KEY (но по умолчанию off, включается PRAGMA)[andrewzuo.com](https://andrewzuo.com/how-to-fix-sqlites-foreign-keys-9d3b0d41d36d#:~:text=SQLite%20foreign%20keys%20are%20not,do%20not%20create%20indexes) – у нас можно по умолчанию включить. CHECK-ограничения будут проверяться на уровне вставки/обновления (парсить выражение и вычислять на уровне отдельной записи). UNIQUE – обеспечивается уникальным индексом (при попытке вставки дубликата индекс вернет ошибку). NOT NULL – проверка в момент вставки тоже. **Triggers** (триггеры) – можно реализовать частично: парсить `CREATE TRIGGER` и сохранять связанное действие (например, SQL-выражение или даже C#-делегат) для выполнения при событии. Это сложный функционал, поэтому в первой версии можно отложить или ограничить (например, только AFTER triggers без FOR EACH STATEMENT, а только FOR EACH ROW). Но для паритета с SQLite хорошо бы хотя бы простые триггеры иметь.

 

**Встроенные функции.** Реализуем набор стандартных SQL-функций: строковые (LENGTH, SUBSTR, UPPER, LOWER), математические (ABS, etc), датавременные (DATE(), datetime() – можно на основе DateTime.UtcNow), агрегатные (SUM, COUNT, MIN, MAX, AVG). Также функции для JSON, если будем поддерживать JSON-тип – SQLite имеет JSON1 extension с функциями `json_extract`, `json_quote` и т.д.[sqlite.org](https://sqlite.org/json1.html#:~:text=By%20default%2C%20SQLite%20supports%20thirty,can%20be%20used%20to). Можно встроить популярные: `JSON_VALUE`, `JSON_QUERY` или упрощенно хотя бы хранение JSON как текст + функция извлечения. Это не первоочередно, но стоит в дизайне держать возможность.

 

**Планировщик транзакций.** Важный момент – обеспечить целостность видимости данных: т.е. пока транзакция не зафиксирована, её данные не должны увидеть другие. В SQLite это достигается либо копированием страниц в журнал (до изменений) либо WAL. У нас, если WAL, то читатели читают старые страницы, пока writer пишет новые в журнале. Нужно правильно реализовать механизм снапшота для чтения: можно сделать так, что транзакция чтения фиксирует текущее состояние, и если в процессе другие транзакции коммитят, читатель все равно видит консистентную картину на момент начала (snapshot isolation, как в SQLite **SERIALIZABLE** по умолчанию). Это потребует либо копий страниц (как MultiVersion Concurrency Control, MVCC) либо блокировок на время чтения. Проще, в рамках SQLite-семантики: чтение берет shared-лок, писатель ждёт пока активные читатели завершатся если конфликт, а чтение, начатое раньше, не видит новые данные (потому что writer либо ждет, либо в WAL). Мы так и поступим. То есть **изолированность** – уровень, близкий к **READ SERIALIZABLE**: никакой "грязной" читаемости незакоммиченных данных.

### 3. Интеграция с .NET и EF Core

**ADO.NET провайдер.** Будет создан проект (или класс) реализующий интерфейсы ADO.NET: `DbConnection`, `DbCommand`, `DbDataReader`, `DbTransaction`, `DbParameter` и пр. Назовём их `WitDbConnection`, `WitDbCommand` и т.д. `WitDbConnection` будет содержать объект-сессию движка (например, ссылку на экземпляр класса, управляющего открытым файлом базы и кэшем страниц). При `Open()` – будет открываться файл (или создаваться, если не существует, аналогично SQLite поведение). `WitDbCommand` принимает текст SQL и параметры, при выполнении вызывает движок парсера/исполнителя. `WitDbDataReader` предоставляет построчный доступ к результатам – т.е. внутри будет держать ссылку на итератор результата и при каждом `Read()` получать следующую строку, а методы типа `GetInt32`, `GetString` будут извлекать данные из текущей строки. Также реализация `GetSchemaTable()` и свойств `FieldCount`, `DataTypeName` для совместимости с стандартными средствами. `DbTransaction` – обертка над началом/концом транзакции в движке.

 

Особое внимание – **параметры SQL**. В SQLite параметры обозначаются как `?` или именованные `@param`. Мы реализуем поддержку параметров в `WitDbCommand` (например, заменяя `?` на placeholders или прямо подставляя значения безопасно). Будет класс `WitDbParameter` с свойствами `DbType`, `Value` и т.д. При выполнении команда передаст параметры в движок, который свяжет их с плейсхолдерами в скомпилированном запросе. Возможно, стоит реализовать кеширование подготовленных запросов: например, если одна и та же SQL строка выполняется часто, парсить ее один раз и хранить AST/план, а при повторном выполнении с другими параметрами – просто привязывать новые значения. SQLite делает prepared statements и параметризацию. Мы тоже это сделаем для эффективности.

 

**Entity Framework Core провайдер.** EF Core взаимодействует с БД через провайдеры, специфичные для СУБД. Нужно создать библиотеку `Microsoft.EntityFrameworkCore.WitDB` (или под другим неймспейсом), реализующую необходимые классы: например, `WitDbOptionsExtension`, `WitDbProviderServices`, `WitDbMigrationsSqlGenerator`, `WitDbQuerySqlGenerator` и пр. К счастью, EF Core уже имеет реализованный провайдер для SQLite[learn.microsoft.com](https://learn.microsoft.com/en-us/dotnet/standard/data/sqlite/#:~:text=Microsoft,on%20top%20of%20this%20library), на который можно ориентироваться. Возможно, мы сможем унаследоваться от некоторых базовых классов (SQLite-провайдер является частным внутри EFCore, но можно посмотреть его исходники). Главные задачи EF-провайдера:

- **Выдача корректного SQL**. EF Core генерирует LINQ-Expressions и передаёт на генерацию SQL. Для WitDB можно использовать очень похожий синтаксис на SQLite, т.к. у нас схожая диалектика. Например, EF-запрос `.Where(x => x.Name.StartsWith("A"))` превратится в `WHERE Name LIKE 'A%'`. Большинство такого перевода EF Core делает сама, провайдер лишь настраивает нюансы (например, как экранировать именя параметров, поддерживает ли `LIMIT` (да, SQLite/WitDB поддерживает) или нужно `TOP`, и т.д.). Мы должны убедиться, что SQL, генерируемый EF Core, понятен нашему движку. Это может потребовать небольших доработок генератора: к примеру, в SQLite нет отдельного типа `DateTime`, EF может генерировать функции для дат – нам либо реализовать их, либо изменить перевод. Но в целом, стремимся, чтобы **любой LINQ запрос, работавший с SQLite через Microsoft.Data.Sqlite, так же работал с WitDB**.
    
- **Миграции.** EF Core миграции создают DDL команды. Например, `AddColumn`, `DropColumn`, `AddForeignKey` – SQLite-провайдер EF Core иногда должен обходить ограничения SQLite. Мы постараемся сделать эти операции **нативно**: DropColumn — выполним внутри WitDB (удалим из схемы и данных), AddForeignKey – просто добавим constraint в метаданные. То есть миграции будут прямолинейно выполняться. Провайдер должен уметь генерировать SQL для операций миграции. В EF Core это настраивается через `MigrationSqlGenerator`. Мы разработаем свою реализацию, генерирующую соответствующие SQL DDL под WitDB. Если какие-то операции не поддерживаются напрямую, возможно, сделаем так же как SQLite-провайдер: например, при переименовании столбца – EF Core для SQLite создает новую таблицу без старого столбца, копирует данные, удаляет старую, переименует новую. Мы можем упростить, если реализуем `ALTER TABLE RENAME COLUMN` напрямую в WitDB, тогда миграция станет одной командой.
    
- **Тестирование с EF.** Необходимо прогнать полный набор тестов EF Core (можно взять Microsoft.EntityFrameworkCore.Specification.Tests) на провайдере WitDB, чтобы убедиться в корректной работе: отслеживание изменений, сохранение связанных сущностей, lazy/eager loading, LINQ-группировки и пр. По результатам, возможно, придётся кое-что доработать либо в движке (например, добавить поддержку функции `LIKE` с ESCAPE, или правильную семантику `DateTime.Kind`). Эти мелочи выплывут в тестах.
    
- **Поддержка EF Core Analyzer/Scaffold.** Опционально, но желательно: чтобы можно было выполнять `dotnet ef dbcontext scaffold` на файл WitDB (реверс-инжиниринг) – придётся реализовать `IProviderConfiguration` чтобы tooling распознавал наш провайдер по строке подключения. Также QueryableExtensions типа `UseWitDB(string connectionString)` в пакете, чтобы удобно было настраивать в `OnConfiguring`.
    

### 4. Современные технологии и оптимизации реализации

Как проект на .NET 8/9, WitDB будет использовать новейшие возможности платформы для достижения высокой эффективности:

- **Span<T> и побайтная обработка.** Как отмечалось, основной акцент – минимизация лишних аллокаций. Все низкоуровневые операции (чтение/запись страниц, парсинг полей) будут работать через `Span<byte>`/`ReadOnlySpan<byte>`. Например, чтобы прочитать int из буфера страницы, мы не будем создавать массив из 4 байт – вместо этого с помощью `BinaryPrimitives.ReadInt32LittleEndian(span.Slice(offset))` сразу получим значение. Работа со Span позволяет JIT оптимизировать копирования и избегать границ массива.
    
- **Array Pooling.** Для буферов ввода-вывода (например, чтение страницы с диска) применим `ArrayPool<byte>.Shared`. Это предотвратит постоянное выделение новых byte[4096] при каждом обращении – вместо этого будем арендовать из пула и возвращать после использования. Таким образом снижается фрагментация памяти и нагрузка на сборщик.
    
- **Предвыделенные структуры для значений.** Реализуем структуру `RecordBuffer` или `RowBuffer`, которая хранит значения столбцов текущей записи, используя поля фиксированных типов или массив значений `SqlValue`. Это позволит избегать создания множества мелких объектов-оберток при чтении строк – DataReader сможет возвращать данные напрямую из этой структуры. Например, `WitDbDataReader.GetInt32(i)` просто возьмёт значение из массива значений текущей строки, уже раскодированного.
    
- **Параллелизм при выполнении запросов.** В перспективе, для сложных запросов (например, большие сканирования или join) можно использовать возможности TPL. Например, если запрос выполняет последовательный полный обход таблицы с фильтрацией, а таблица очень большая, можно задействовать несколько потоков для разных сегментов таблицы (разбивка по диапазонам ключей). Однако, встроенные СУБД редко применяют такой внутрипроцессный параллелизм, дабы не усложнять. На первом этапе мы этого делать не будем. Но оставим архитектурно возможность – например, в планировщике предусмотреть шаг, который мог бы распределить работу по thread pool (с учётом, что при embedded использовании, выделение нескольких потоков под один запрос спорно). Скорее, более актуально – **асинхронный API**.
    
- **Async/await поддержка.** ADO.NET интерфейсы имеют методы `ExecuteReaderAsync`, `ExecuteNonQueryAsync` и т.п. Мы реализуем их полностью асинхронными. То есть, если требуется чтение с диска, будем использовать `Stream.ReadAsync`. Если нужно подождать блокировку, не будем блокировать поток, а использовать асинхронный `SemaphoreSlim.WaitAsync`. Это важно для ASP.NET Core приложений (чтобы не подвешивать потоки) и вообще хорошая практика. SQLite через `Microsoft.Data.Sqlite` реализован синхронно внутри (async там фактически обёртка), а мы можем сделать **настоящий async**, поскольку у нас полный контроль. Таким образом, WitDB будет отлично вписываться в масштабируемые серверные приложения на C#.
    
- **JIT/AOT оптимизации.** Используя .NET 8/9, мы можем задействовать профилируемую компиляцию (PGO) – если какой-то метод критичен (например, внутренняя функция поиска в странице), он со временем оптимизируется JIT'ом. При AoT-компиляции (WASM, iOS) стоит обеспечивать наличие необходимых подсказок (например, `MethodImpl(AggressiveInlining)` на маленьких методах) для более эффективной компиляции. Кроме того, .NET позволяет использовать аппаратные инструкции (SIMD) через `System.Numerics.Vector`. Возможное применение: ускорение сравнения байтовых ключей, когда нужно быстро сравнить две последовательности bytes (можно 16 байт за раз через Vector128). Или поиск по строке. Эти микро-оптимизации не в первую очередь, но в критичных местах можно измерить и внедрить.
    
- **Локальность и кеш-линия.** Нужно стараться хранить метаданные так, чтобы часто используемые вместе данные лежали рядом в памяти (например, struct с полями вместо классов, чтобы были contiguous). Это улучшит кеш-поведение на уровне CPU. Например, хранение узла B-дерева как структуры фиксированного размера, содержащей массив ключей и массив ссылок – можно разместить его на unmanaged памяти или stackalloc при поиске. Возможно, иногда будем использовать `stackalloc` для короткоживущих структур (например, буфер составления ключа) – это тоже снижает нагрузку на куче.
    
- **Логирование и диагностика.** Встроим в провайдер поддержку логирования (например, через стандартный `ILogger` EF Core или собственный callback) для отладки – чтобы можно было включить вывод выполняемых SQL, время выполнения запросов, статистику кеш-хитов, и др. Современные приложения ценят возможность наблюдаемости. Мы можем, например, интегрировать счетчики (`System.Diagnostics.Metrics`) – количество транзакций, средний размер страниц в кэше, и т.п., чтобы разработчики могли мониторить поведение WitDB.
    

### 5. Этапы разработки (пошаговый план)

Ниже предлагается поэтапный план реализации WitDB, отражающий приоритеты:

1. **Проектирование формата и базовых структур.** Сначала оформить спецификацию формата файла: размер страницы, структура заголовка, схемы кодирования. На этом этапе можно создать модуль, умеющий создавать новый файл базы, открывать существующий, читать/писать страницы, управлять простейшим каталогом таблиц. Реализовать базовые классы: Page, Pager, FileStorage. Написать юнит-тесты на чтение-запись страниц, рост файла и т.п.
    
2. **Реализация B-дерева для таблиц и индексов.** Этот этап наиболее алгоритмически сложный. Нужно реализовать вставку, поиск, удаление в B+tree. Начать можно с таблиц без вторичных индексов: т.е. ключ – rowid, значение – запись. Проверить, что можно вставлять много записей, они читаются обратно, работают переходы на новые страницы при переполнении. Затем добавить поддержку вторичных индексов: у индекса ключ составной (значение поля + rowid как тайбрейкер), значение пустое или ссылка. Убедиться, что поиск по индексу выдает соответствующие rowid, которые можно потом искать в таблице.
    
3. **SQL-запросы: простой исполнитель.** Реализовать минимально необходимый парсер (вначале можно поддержать только одну команду `SELECT` с простыми условиями) и исполняющую часть. Цель – получить первый работающий запрос "end-to-end": например, `SELECT * FROM MyTable WHERE Column = 123`. Для этого:
    
    - Создать структуру хранения схемы (список столбцов таблицы, их типы).
        
    - Вставку данных пока можно делать через низкоуровневый API (например, метод Insert(table, values)).
        
    - Реализовать TableScanIterator и FilterIterator, чтобы выборка работала.
        
    - Вывести результат через временный DataReader заглушку.
        
    - Как только SELECT заработал, добавить `INSERT`, `UPDATE`, `DELETE` обработку (они проще – достаточно найти нужную таблицу и вызвать методы вставки/удаления).
        
    
    Протестировать сценарий: создать таблицу, вставить несколько строк, выбрать их по условию – и получить ожидаемые результаты.
    
4. **Расширение SQL и выражений.** Усложнить парсер: поддержать JOIN (nested loop), агрегатные функции (`GROUP BY`), ORDER BY (потребуется сортировка – можно на небольших наборах через LINQ OrderBy или написать QuickSort для итератора). Добавить обработку более сложных выражений в WHERE (с комбинацией AND/OR, скобки – построить дерево выражений и вычислитель для него). Добавить встроенные функции (например, сначала LENGTH, UPPER, etc., которые можно вызвать в выражениях).
    
5. **Транзакции и журналирование.** Внедрить механизм транзакций: создать класс Transaction, добавить в движок методы BeginTransaction, Commit, Rollback. На этом этапе реализовать либо запись копий страниц в rollback journal, либо механизм WAL. Например, начать с более простого: при начале транзакции помечаем какие страницы изменяем, перед изменением сохраняем их оригинал во временный файл (journal). Если транзакция откатывается или при сбое – просто выкидываем изменения и восстанавливаем сохранённые страницы. Если коммит – либо перезаписываем страницы и удаляем журнал. WAL можно реализовать следующим шагом. Главное – после этого шага можно гарантировать целостность при сбоях.
    
6. **Конкурентность и блокировки.** Реализовать управление многопоточным доступом: ввести глобальный объект блокировки или несколько уровней. Например, использовать `ReaderWriterLockSlim` для разграничения concurrent read/write в одном процессе. При BeginTransaction(чтения) – брать шаред-лочок, при BeginTransaction(записи) – эксклюзивный. Учесть, что если запись не может захватить (есть активные чтения), можно ждать (с таймаутом). Для межпроцессного – использовать файловую блокировку: при открытии файла лочить весь файл (или отдельный регион) в режиме разделяемом или эксклюзивном. Проверить на практике, что два процесса не портят файл (это сложно тестировать, можно отложить проверку). На этом этапе можно также реализовать WAL вместо rollback: будет другой механизм записи – вместо inplace-обновления страницы пишем новую копию в WAL-файл. Требуется немного перестроить логику коммита/чтения (читающие транзакции должны видеть либо старые, либо новые версии страниц). Это сложнее, можно оставить WAL на потом, но лучше реализовать сейчас ради соответствия требованию конкурентности.
    
7. **EF Core провайдер и ADO.NET API.** Когда движок в целом стабильно выполняет SQL, можно написать ADO.NET-обёртки. Реализовать WitDbConnection/Open, WitDbCommand/ExecuteReader etc., пока без оптимизации (каждый Execute – парсит SQL заново). Написать первые интеграционные тесты через ADO: например, подключиться, выполнить `CREATE TABLE` через ExecuteNonQuery, затем `INSERT`, затем `SELECT` через ExecuteReader и проверить получение данных. Отладить типы данных – чтобы `GetInt32` возвращал именно Int32, `GetString` – .NET string, и т.д.
    
8. **Разработка EF Core провайдера.** Настроить проект под EF Core. Возможно, начать с копирования существующего SQLite-провайдера и замены специфики. Нужно реализовать классы:
    
    - `WitDbConnection` (наследник `RelationalConnection` EF),
        
    - `WitDbConventionSetBuilder` (если нужны специальные правила именования, можно позаимствовать SQLiteConventionSetBuilder, который включает ограничения типа MaxLength для keys),
        
    - `WitDbMigrationSqlGenerator` (породить от SqlServer или SQLite генератора и подправить),
        
    - `WitDbQuerySqlGenerator` (наследовать от `QuerySqlGenerator`).
        
    - Зарегистрировать provider invariant name, например "WitDB".
        
    
    Затем попробовать выполнить простейший EF Core сценарий:
    
    `options.UseWitDB("Data Source=test.witdb"); context.SaveChanges(); var list = context.MyEntities.Where(x=>x.Field == some).ToList();`
    
    И отладить, какие SQL генерируются, проходят ли они. Это, вероятно, выявит какие-то незакрытые возможности (например, EF может сгенерировать `PRAGMA foreign_keys=ON` при открытии – хорошо бы поддержать его как noop или действие).
    
9. **Тонкая доводка и оптимизация.** Когда функционально все части есть, заняться профилированием и улучшением скорости. Запустить набор бенчмарков: вставка N записей, выборка по первичному ключу, сканирование всей таблицы. Сравнить с SQLite в аналогичных условиях[tomaskohl.com](https://tomaskohl.com/code/2020-04-07/trying-out-litedb/#:~:text=SQLite%20has%20long%20been%20the,well%20when%20used%20as%20intended). Найти узкие места: может, парсер слишком медленный – можно ввести кеш prepared statements. Может, операция сравнения строк – подключить Span/Vector оптимизацию. Также оптимизировать аллокации: профилировать в .NET Memory Profiler, убрать лишние `new`. Обратить внимание на многопоточность: протестировать сценарий, когда 10 потоков читают, 1 пишет – приложение не должно “падать” или заходить в deadlock. При необходимости настроить более гранулярные блокировки (например, отдельный lock на каждую таблицу для параллельных транзакций в разные таблицы – но SQLite так не делает, он проще с глобальной блокировкой). Возможно, реализовать **page-level locking** для concurrent writer (MVCC), но это очень сложно – может остаться на будущее.
    
10. **Дополнительные возможности.** Реализовать отложенные ранее фишки: триггеры, представления (VIEW – фактически сохраненные запросы; для их поддержки в парсере несложно, но при изменении базовых таблиц никаких обновлений view не нужно, достаточно позволить делать SELECT из view, подставляя определение). Пользовательские функции – сделать API `WitDbConnection.CreateFunction(name, func<...>)`. Подумать о **full-text search**: SQLite предлагает FTS5 модуль. Для WitDB можно интегрировать простой полнотекстовый поиск как дополнительный индекс (например, хранить словарь). Это отдельный подсистем, возможно плагин в будущем. Сейчас не обязательно.
    
11. **Документация и выпуск.** Подготовить читаемую документацию: какие SQL поддержаны, на что есть ограничения (в первой версии может не быть, скажем, сложных оконных функций или recursive CTE). Описать, как подключаться, как использовать с EF Core (пример OnConfiguring). Выложить проект на GitHub/NuGet. Получить фидбэк от сообщества, далее по итерациям улучшать.

В итоге, **WitDB** должен предоставить сообществу .NET полноценную альтернативу SQLite: управляемую, совместимую, простую во внедрении. Соблюдая перечисленные требования и следуя намеченному плану, можно создать продукт, сочетающий надежность и удобство SQLite (прошедшего проверку временем) с преимуществами экосистемы .NET[tomaskohl.com](https://tomaskohl.com/code/2020-04-07/trying-out-litedb/#:~:text=Just%20like%20SQLite%2C%20LiteDB%3A)[geeksforgeeks.org](https://www.geeksforgeeks.org/sql/introduction-to-sqlite/#:~:text=expressions,platform%20disk%20file). Эта СУБД сможет работать в любых .NET-приложениях – от десктопа до браузера – и даст разработчикам гибкость и независимость от платформы, оставаясь при этом знакомой по возможностям и поведению. Если SQLite поддерживает какую-то возможность, WitDB также будет стремиться ее поддерживать[geeksforgeeks.org](https://www.geeksforgeeks.org/sql/introduction-to-sqlite/#:~:text=2,the%20direct%20file%20system%20I%2FO), но при этом проект открывает двери для инноваций, используя лучшие оптимизации современного .NET. Такой подход позволит WitDB занять достойное место в инструментарии .NET-разработчика, удовлетворяя потребности в локальном хранилище данных с богатыми возможностями и простотой использования.
